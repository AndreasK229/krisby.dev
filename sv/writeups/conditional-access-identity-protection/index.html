Conditional Access + Identity Protection Lab

By Andreas Krisby 2026-05-25 3 min read

Detta labbet byggdes för att förstå hur riskbaserad Conditional Access och Identity Protection fungerar i Microsoft Entra ID.

Målet var att:

  • Skapa riskbaserade Conditional Access-policies
  • Tvinga MFA vid risky sign-ins
  • Förstå user risk och sign-in risk
  • Testa Identity Protection i praktiken
  • Analysera risky users och risky sign-ins

Arkitektur

User Sign-In → Identity Protection → Risk Detection → Conditional Access → MFA

Steg 1 — Förberedelser

Verifiera att tenant har rätt funktioner aktiverade:

  • Microsoft Entra ID P2
  • Identity Protection
  • Conditional Access

Skapa ett separat testkonto och tilldela Entra ID P2-licens:

test-risk-user@tenant.onmicrosoft.com

Steg 2 — Konfigurera MFA

Logga in med testkontot och registrera MFA via Microsoft Authenticator. Verifiera att MFA fungerar korrekt innan nästa steg.

Steg 3 — Skapa Conditional Access Policy

Entra Admin Center → Protection → Conditional Access → Policies → New Policy

Policynamn:

CA - Require MFA for Risky Sign-ins

Steg 4 — Konfigurera users

Under Assignments → Users:

  • All users (eller specifikt testkonto)

Break glass- och admin-konton exkluderas från policyn.

Steg 5 — Konfigurera risk conditions

Under Conditions → Sign-in risk:

  • Configure: Yes
  • Välj: Medium and High

Under Conditions → User risk:

  • Configure: Yes
  • Välj: Medium and High

Steg 6 — Konfigurera Access Controls

Under Access controls → Grant:

  • Grant access
  • Require multi-factor authentication

Klicka Select.

Steg 7 — Enable policy

Under Enable policy:

  • Välj: On

Klicka Create.

Steg 8 — Testa risky sign-ins

Olika inloggningsscenarion testades för att trigga Identity Protection:

  • VPN
  • Incognito-läge
  • InPrivate-sessioner
  • Olika browsers
  • Upprepade sign-ins
  • Flera nya sessioner

Målet var att se om Entra började markera användaren eller sign-ins som risky.

Steg 9 — Övervaka Identity Protection

Entra Admin Center → Protection → Identity Protection

Kontrollera:

  • Risky users
  • Risky sign-ins
  • Risk detections

Efter ett tag dök testkontot upp som Risky User.

Steg 10 — Hantera risky user

Öppna användaren under Identity Protection → Risky users. Tillgängliga åtgärder:

Revoke sessions — tvingar användaren att logga in igen.

Confirm user safe — markerar användaren som säker.

Dismiss user risk — tar bort riskstatus manuellt.

Confirm compromised — markerar användaren som komprometterad.

Steg 11 — Testa What If Tool

Conditional Access → What If

Välj:

  • Test user
  • Cloud app
  • Location/sign-in conditions

Verifiera:

  • Vilka policies som appliceras
  • Om MFA triggas
  • Hur Conditional Access påverkar sign-ins

What If Tool är ovärderligt innan utrullning av policies i produktion.

Resultat

Labbet kunde nu:

  • Identifiera risky users
  • Identifiera risky sign-ins
  • Tvinga MFA vid risk
  • Hantera risknivåer dynamiskt
  • Testa riskbaserad Conditional Access

Lärdomar

  • User risk och sign-in risk är två olika saker
  • Conditional Access kan reagera dynamiskt på risknivåer
  • MFA är effektivt skydd vid risky sign-ins
  • Identity Protection kan upptäcka avvikande beteenden
  • What If Tool är viktigt innan utrullning av policies
  • Risky users kan hanteras direkt i Entra
