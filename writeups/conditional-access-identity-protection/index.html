Conditional Access + Identity Protection Lab

By Andreas Krisby 2026-05-25 3 min read

This lab was built to understand how risk-based Conditional Access and Identity Protection work in Microsoft Entra ID.

The goal was to:

  • Create risk-based Conditional Access policies
  • Force MFA on risky sign-ins
  • Understand user risk and sign-in risk
  • Test Identity Protection in practice
  • Analyze risky users and risky sign-ins

Architecture

User Sign-In → Identity Protection → Risk Detection → Conditional Access → MFA

Step 1 — Preparation

Verify that the tenant has the right features enabled:

  • Microsoft Entra ID P2
  • Identity Protection
  • Conditional Access

Create a separate test account and assign Entra ID P2 license:

test-risk-user@tenant.onmicrosoft.com

Step 2 — Configure MFA

Sign in with the test account and register MFA via Microsoft Authenticator. Verify that MFA works correctly before the next step.

Step 3 — Create Conditional Access Policy

Entra Admin Center → Protection → Conditional Access → Policies → New Policy

Policy name:

CA - Require MFA for Risky Sign-ins

Step 4 — Configure users

Under Assignments → Users:

  • All users (or specific test account)

Break glass and admin accounts are excluded from the policy.

Step 5 — Configure risk conditions

Under Conditions → Sign-in risk:

  • Configure: Yes
  • Select: Medium and High

Under Conditions → User risk:

  • Configure: Yes
  • Select: Medium and High

Step 6 — Configure Access Controls

Under Access controls → Grant:

  • Grant access
  • Require multi-factor authentication

Click Select.

Step 7 — Enable policy

Under Enable policy:

  • Select: On

Click Create.

Step 8 — Test risky sign-ins

Different sign-in scenarios were tested to trigger Identity Protection:

  • VPN
  • Incognito mode
  • InPrivate sessions
  • Different browsers
  • Repeated sign-ins
  • Multiple new sessions

The goal was to see if Entra started marking the user or sign-ins as risky.

Step 9 — Monitor Identity Protection

Entra Admin Center → Protection → Identity Protection

Check:

  • Risky users
  • Risky sign-ins
  • Risk detections

After a while, the test account appeared as a Risky User.

Step 10 — Manage risky user

Open the user under Identity Protection → Risky users. Available actions:

Revoke sessions — forces the user to sign in again.

Confirm user safe — marks the user as safe.

Dismiss user risk — removes risk status manually.

Confirm compromised — marks the user as compromised.

Step 11 — Test What If Tool

Conditional Access → What If

Select:

  • Test user
  • Cloud app
  • Location/sign-in conditions

Verify:

  • Which policies are applied
  • Whether MFA is triggered
  • How Conditional Access affects sign-ins

The What If Tool is invaluable before deploying policies in production.

Results

The lab could now:

  • Identify risky users
  • Identify risky sign-ins
  • Force MFA on risk
  • Manage risk levels dynamically
  • Test risk-based Conditional Access

Lessons

  • User risk and sign-in risk are two different things
  • Conditional Access can react dynamically to risk levels
  • MFA is effective protection on risky sign-ins
  • Identity Protection can detect unusual behavior
  • What If Tool is important before deploying policies
  • Risky users can be managed directly in Entra
