JML Monitor Lab

By Andreas Krisby 2026-02-10 3 min read

Built a simple JML monitor for Microsoft Entra ID to understand Joiner/Mover/Leaver patterns, group-based RBAC, how access changes over time, and how to detect access drift. Also to understand how Graph API works in practice.

Step 1 – Create Entra lab

Created users in Microsoft Entra ID:

  • HR users
  • IT users
  • Admin users
  • Production users

Created groups: HR, Admin, Production, Technical administrator VM, VNET TECH.

Added users to different groups to simulate RBAC.

Step 2 – Register app in Entra

  1. Entra ID → App registrations → New registration
  2. Give the app a name, e.g. JML-Monitor
  3. Save the Tenant ID and Client ID after creation

Step 3 – Create client secret

  1. Certificates & secrets → New client secret
  2. Save the secret value immediately (only shown once)

Step 4 – API Permissions

Add Microsoft Graph permissions:

  • User.Read.All
  • Group.Read.All
  • Directory.Read.All
  • AuditLog.Read.All

Grant admin consent.

Step 5 – Install Python

pip install requests

Step 6 – Create Python script

Create a file get_token.py that handles authentication and API calls.

Note: The Python script skeleton was written with AI assistance (Claude). The logic and architecture — how Graph API integrates, which permissions are required, how to detect access drift — are based on my understanding of identity systems and Entra ID. AI assistance was a tool to convert the logic into working code.

Step 7 – Auth to Entra

Authentication to Microsoft Identity Platform with client credentials:

url = f"https://login.microsoftonline.com/{tenant_id}/oauth2/v2.0/token"

data = {
    "grant_type": "client_credentials",
    "client_id": client_id,
    "client_secret": client_secret,
    "scope": "https://graph.microsoft.com/.default"
}

response = requests.post(url, data=data)
token = response.json()["access_token"]

Step 8 – Fetch users

response = requests.get(
    "https://graph.microsoft.com/v1.0/users",
    headers={"Authorization": f"Bearer {token}"}
)
users = response.json()

Returns: userPrincipalName, id, displayName.

Step 9 – Loop users

for user in users["value"]:
    name = user["displayName"]
    upn  = user["userPrincipalName"]
    uid  = user["id"]

Step 10 – Fetch groups per user

url = f"https://graph.microsoft.com/v1.0/users/{user_id}/memberOf"
groups = requests.get(url, headers=headers).json()

Step 11 – Loop groups

user_groups = []
for g in groups.get("value", []):
    user_groups.append(g["displayName"])

Step 12 – Create detection logic

If-logic to flag risks:

  • Admin access
  • HR + Admin (SoD conflict)
  • Disabled account with active access
  • New groups since last scan
  • Removed access
  • Access remaining after offboarding

Step 13 – Create snapshot

Save current state to state.json:

{
  "user-id-123": {
    "name": "Anna HR",
    "groups": ["HR", "GG_HR_Read"]
  }
}

Step 14 – Compare state

On next run:

  1. Read old state.json
  2. Fetch new data from Graph
  3. Compare: new groups, removed groups, changes

Step 15 – Detection example

The script could show:

⚠️  NEW ACCESS       Anna HR      → Admin
🚨  REMOVED ACCESS   Johan IT     ← Production
🚨  RISK: Admin access           Sara IT
🚨  RISK: SoD conflict           Erik HR (HR + Admin)

Step 16 – Secure secret

Realized the secret had been exposed. Actions:

  • Created new secret
  • Moved secret to .env
  • Added .env to .gitignore

Step 17 – Results

The monitor could now:

  • Track group changes
  • Detect access drift
  • Show Mover/Leaver scenarios
  • Analyze RBAC via groups
  • Compare history over time

Lessons

  • RBAC becomes clearer via groups
  • History is needed to detect changes
  • Access drift is only visible when comparing
  • JML becomes clear in practice
  • Graph API provides valuable data
  • Security is much about logic and structure
