Entra ID + Cloudflare Access + SCIM

By Andreas Krisby 2026-05-23 3 min read

This lab was set up to understand how federation, provisioning and Zero Trust work in practice between Entra ID and Cloudflare Access.

The goal was to:

  • Use Entra ID as Identity Provider
  • Set up OIDC authentication
  • Set up SCIM provisioning
  • Automate sync of users and groups
  • Test group-based access via Cloudflare Access

Architecture

Entra ID → OIDC → Cloudflare Access

Entra ID → SCIM → Cloudflare

Step 1 — Create App Registration in Entra ID

The first step was to create an App Registration in Entra ID.

  • Entra Admin Center → App registrations → New registration
  • Name: Cloudflare Access
  • Supported account types: Single tenant

The Redirect URI was left empty at this step.

Step 2 — Create Identity Provider in Cloudflare

After App Registration, the Identity Provider was created in Cloudflare Zero Trust.

  • Cloudflare Dashboard → Zero Trust → Settings → Authentication → Add new → Azure AD

Cloudflare then generated the redirect URI needed in Entra ID:

https://team.cloudflareaccess.com/cdn-cgi/access/callback

This URI was copied back to the App Registration in Entra.

Step 3 — Configure Redirect URI

In Entra: App registrations → Cloudflare Access → Authentication → Add platform → Web

Redirect URI:

https://team.cloudflareaccess.com/cdn-cgi/access/callback

I got stuck here first because the redirect URI must match exactly between systems.

Step 4 — API Permissions

Add Microsoft Graph — Delegated permissions:

  • openid
  • profile
  • email
  • User.Read
  • GroupMember.Read.All

Then: Grant admin consent

Without admin consent, some parts didn’t work correctly.

Step 5 — Create Client Secret

Certificates & secrets → New client secret

Cloudflare needed:

  • Application (client) ID
  • Directory (tenant) ID
  • Client Secret VALUE

It’s VALUE that should be used — not Secret ID. This caused auth problems before I found the error.

Step 6 — Connect Entra ID to Cloudflare

In Cloudflare Zero Trust → Settings → Authentication → Azure AD:

  • Client ID
  • Tenant ID
  • Client Secret

After this, login via Entra ID to Cloudflare Access worked.

Step 7 — Test Authentication Flow

Created a subdomain test.krisby.dev and an Access Application:

Zero Trust → Access → Applications → Add application → Self-hosted → test.krisby.dev

When the page was opened:

  1. Cloudflare redirected to Entra ID
  2. Authentication occurred via Entra
  3. Token was sent back to Cloudflare
  4. Access was granted after successful auth

Step 8 — Set up SCIM Provisioning

In Cloudflare: Settings → SCIM

Cloudflare generated SCIM Endpoint and SCIM Token.

In Entra: Enterprise Applications → Cloudflare Access → Provisioning

Provisioning mode: Automatic

SCIM endpoint and token were pasted in.

Step 9 — Group-based Provisioning

Groups were created in Entra ID and assigned to Enterprise Application. After sync:

  • Users sync automatically
  • Groups sync automatically
  • Access is managed group-based

Provisioning and deprovisioning then worked automatically via SCIM.

Lessons

  • SCIM greatly simplifies lifecycle management
  • Group-based access saves a lot of manual work
  • Redirect URI must match exactly
  • Client Secret VALUE and Secret ID are not the same thing
  • It’s much easier to understand auth flows when systems run live
